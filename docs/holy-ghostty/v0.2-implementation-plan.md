# Holy Ghostty v0.2 Implementation Plan

Last updated: 2026-04-15

This document translates the public roadmap into a concrete implementation plan for `v0.2`.

`v0.2` is the foundation release. Its job is not to make Holy Ghostty broader. Its job is to make Holy Ghostty internally trustworthy enough to support everything that comes after it.

## v0.2 Outcome

By the end of `v0.2`, Holy Ghostty should have:

- a durable SQLite-backed persistence layer
- an append-oriented event ledger
- a real split between UI state, supervision, and persistence
- archive/history backed by durable evented state instead of JSON snapshots
- a migration path from the current `v0.1` JSON snapshot store
- a versioned read-only compatibility surface for future external consumers

`v0.2` is complete when the app can be killed mid-session, restarted, migrated, and resumed without losing the operational memory that the product depends on.

## What v0.2 Is Not

`v0.2` is not the release for:

- cost and budget intelligence
- grid mode, focus mode, or diff mode
- full external task integration
- broad UI redesign

Those all depend on `v0.2` shipping first.

## Current Architecture Pressure Points

The current `v0.1` architecture is strong enough to prove the product, but weak in the exact places a real platform cannot stay weak.

Current pressure points:

### 1. JSON snapshot persistence

Current persistence lives in:

- `macos/Sources/HolyGhostty/Persistence/HolyWorkspacePersistence.swift`

Current problems:

- only snapshot-oriented
- no event history
- no migration system
- no durable alert ledger
- weak crash recovery story

### 2. Monolithic workspace store

Current orchestration lives in:

- `macos/Sources/HolyGhostty/Workspace/HolyWorkspaceStore.swift`

Current problems:

- store, supervisor, persistence, archive, alerting, and draft evaluation are mixed together
- lifecycle responsibilities are hard to isolate
- future telemetry and budgeting would make this file worse

### 3. Event-poor internal model

The product already behaves like a control plane, but it still stores too much as current state and too little as durable event history.

Current missing layer:

- session event timeline
- alert history
- git snapshot history
- lifecycle replay semantics

## Target v0.2 Architecture

The `v0.2` shape should be:

### UI layer

- `HolyWorkspaceStore`
  - remains the SwiftUI-facing observable object
  - owns presentation state
  - stops being the long-term source of lifecycle truth

### Supervision layer

- `HolySessionSupervisor`
  - launch policy
  - lifecycle transitions
  - session registration
  - session archive/relaunch logic
  - restart and recovery orchestration

### State layer

- `HolySessionStore`
  - observable in-memory session state for the UI
  - selected session
  - archive list
  - templates
  - derived coordination cache

### Persistence layer

- `HolyDatabase`
  - SQLite connection management
  - migration execution
  - transactional boundaries
- `HolyEventRepository`
  - append/read session events
  - read current durable state projections
- `HolyTemplateRepository`
  - template CRUD
- `HolyArchiveRepository`
  - archive queries backed by the DB

### Service layer

- `HolyMigrationService`
  - imports `v0.1` JSON state into SQLite on first run
- `HolyAlertCenter`
  - consumes domain events and decides when/how to notify

## Database Strategy

Use system SQLite via `SQLite3` rather than introducing a third-party persistence dependency in `v0.2`.

Reasons:

- macOS-native dependency surface stays small
- schema and migration control remains explicit
- event-ledger needs are modest enough for a direct wrapper
- easier to keep the persistence layer understandable during the architecture split

Database mode:

- WAL journaling
- foreign keys enabled
- explicit `user_version` migrations

Database location:

```text
~/Library/Application Support/<bundle-id>/HolyGhostty/holy-ghostty.sqlite3
```

Companion artifacts:

- `holy-ghostty.sqlite3-wal`
- `holy-ghostty.sqlite3-shm`
- migration/import logs as needed

### Interoperability rule

Holy Ghostty's database is Holy Ghostty's source of truth. It is not a shared multi-app database.

To support future interoperability, `v0.2` should expose versioned read-only compatibility views for external consumers rather than encouraging direct coupling to internal tables.

## Proposed v0.2 Schema

The schema should separate durable current state from append-only history.

### 1. `app_state`

Purpose:

- small durable key/value state for UI restore

Columns:

- `key TEXT PRIMARY KEY`
- `value_json TEXT NOT NULL`
- `updated_at TEXT NOT NULL`

Examples:

- selected session ID
- selected archive ID
- migration version metadata

### 2. `sessions`

Purpose:

- canonical durable session identity

Columns:

- `id TEXT PRIMARY KEY`
- `title TEXT NOT NULL`
- `runtime TEXT NOT NULL`
- `mission TEXT`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`
- `archived_at TEXT`
- `launch_spec_json TEXT NOT NULL`
- `ownership_json TEXT`
- `latest_preview_text TEXT`
- `resume_metadata_json TEXT`
- `latest_phase TEXT`
- `latest_attention TEXT`
- `latest_signal_json TEXT`
- `latest_git_snapshot_id INTEGER`
- `latest_command_telemetry_json TEXT`

Notes:

- `launch_spec_json` preserves the exact launch contract
- the `latest_*` columns are projections for fast UI reads
- `resume_metadata_json` ensures resume behavior is not trapped inside UI-only code
- event history remains authoritative for transitions

### 3. `session_events`

Purpose:

- append-only ledger of session lifecycle and domain events

Columns:

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `session_id TEXT NOT NULL`
- `sequence INTEGER NOT NULL`
- `occurred_at TEXT NOT NULL`
- `event_type TEXT NOT NULL`
- `phase TEXT`
- `attention TEXT`
- `payload_json TEXT`

Indexes:

- `(session_id, sequence)` unique
- `(session_id, occurred_at)`
- `(event_type, occurred_at)`

Event examples:

- session_created
- session_started
- session_selected
- session_waiting_input
- session_failed
- session_completed
- session_archived
- session_relaunched
- command_started
- command_finished
- ownership_drifted
- blocking_conflict_detected
- alert_emitted

### 4. `git_snapshots`

Purpose:

- durable git state history instead of only the latest snapshot in memory

Columns:

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `session_id TEXT NOT NULL`
- `captured_at TEXT NOT NULL`
- `repository_root TEXT`
- `worktree_path TEXT`
- `common_git_directory TEXT`
- `branch TEXT`
- `upstream_branch TEXT`
- `is_detached_head INTEGER NOT NULL`
- `ahead_count INTEGER NOT NULL`
- `behind_count INTEGER NOT NULL`
- `staged_count INTEGER NOT NULL`
- `unstaged_count INTEGER NOT NULL`
- `untracked_count INTEGER NOT NULL`
- `conflicted_count INTEGER NOT NULL`
- `changed_files_json TEXT NOT NULL`

### 5. `templates`

Purpose:

- durable template storage independent of the current snapshot format

Columns:

- `id TEXT PRIMARY KEY`
- `name TEXT NOT NULL`
- `summary TEXT`
- `launch_spec_json TEXT NOT NULL`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`

### 6. `alerts`

Purpose:

- durable alert history and delivery record

Columns:

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `session_id TEXT`
- `session_event_id INTEGER`
- `alert_type TEXT NOT NULL`
- `severity TEXT NOT NULL`
- `title TEXT NOT NULL`
- `body TEXT NOT NULL`
- `delivered_at TEXT NOT NULL`
- `acknowledged_at TEXT`
- `payload_json TEXT`

### 7. Versioned compatibility views

Purpose:

- expose a stable read model for future external consumers without freezing internal table design

Initial target views:

- `agent_sessions_sessions_v1`
- `agent_sessions_resume_targets_v1`
- `agent_sessions_events_v1`

The detailed contract lives in:

- `docs/holy-ghostty/agent-sessions-interoperability.md`

## Domain Event Model

The event model is the core design decision in `v0.2`.

The product should record domain events, not just UI mutations.

Minimum event families:

### Lifecycle events

- session_created
- session_started
- session_activated
- session_completed
- session_failed
- session_archived
- session_deleted
- session_relaunched

### Runtime events

- prompt_ready
- waiting_for_input
- command_started
- command_finished
- signal_changed

### Coordination events

- shared_worktree_detected
- shared_branch_detected
- overlapping_files_detected
- ownership_drift_detected

### Persistence/alert events

- state_restored
- alert_delivered
- archive_materialized

## Supervisor Split

The intended split is:

### `HolyWorkspaceStore`

Keeps:

- sheet presentation
- selected IDs
- view-friendly derived state
- routing of user intents into services

Loses:

- direct restore logic
- direct archive persistence
- direct launch preparation orchestration
- direct alert delivery

### `HolySessionSupervisor`

Owns:

- create / start / archive / relaunch session flows
- wiring live `HolySession` instances to event emission
- replaying persistent state into live session objects
- lifecycle state transitions

### `HolySessionStore`

Owns:

- active sessions
- archived session list
- templates list
- selected IDs as observable state
- coordination projections

### Repository layer

Owns:

- read/write queries
- transactions
- projection updates

## Migration Plan

`v0.2` must not strand `v0.1` users.

Migration strategy:

### Step 1. Introduce SQLite alongside JSON

- create DB if missing
- keep current JSON loader in place temporarily
- build import path from `HolyWorkspaceSnapshot`

### Step 2. First-run import

If the DB is empty and `workspace-state.json` exists:

- load JSON snapshot
- import sessions
- import templates
- import archived sessions
- write migration marker into `app_state`
- preserve the original JSON file until import is verified

### Step 3. Switch reads to DB

- restore from DB first
- fall back to JSON import only when DB is absent or empty

### Step 4. Remove JSON as active source of truth

- JSON persistence stops being the main path
- JSON file can remain as a last-resort migration artifact for one release window

## Milestones

### Milestone A: Database Scaffold

Target:

- DB connection wrapper
- migration runner
- schema v1
- smoke tests for open/migrate/query

Tasks:

- add `HolyDatabasePaths`
- add `HolyDatabase`
- add `HolyDatabaseMigrator`
- create schema v1
- create versioned compatibility views
- test DB creation in Application Support

### Milestone B: Import and Dual Read

Target:

- import `v0.1` JSON state into DB

Tasks:

- create `HolyMigrationService`
- map `HolyWorkspaceSnapshot` into DB rows/events
- switch restore to DB-first
- preserve fallback import path

### Milestone C: Session Event Ledger

Target:

- real event append path for lifecycle and archive flows

Tasks:

- define `HolySessionEvent`
- append events on create/archive/relaunch/selection/state transitions
- persist git snapshots to their own table
- add event timeline queries
- keep compatibility views projecting from durable state rather than ad hoc UI state

### Milestone D: Supervisor Split

Target:

- move lifecycle orchestration out of `HolyWorkspaceStore`

Tasks:

- create `HolySessionSupervisor`
- move session creation/archive/relaunch flows into supervisor
- move alert triggers behind event consumption
- reduce workspace store to view orchestration

### Milestone E: History and Archive Migration

Target:

- archive/history UI backed by DB queries and event projections

Tasks:

- archive queries no longer read from JSON snapshot state
- history view uses durable records and event timeline
- selected archive restore works across relaunch and restart

## First Code Slice

The first slice should be small and irreversible in the right direction.

Start with:

### Slice 1: Persistence Infrastructure Only

Create:

- `HolyDatabasePaths.swift`
- `HolyDatabase.swift`
- `HolyDatabaseMigrator.swift`
- `HolyDatabaseModels.swift`

Implement:

- SQLite open/close
- WAL mode
- schema v1 migration
- DB path creation

Do not yet:

- change UI
- change archive/history flow
- remove JSON persistence

Acceptance for slice 1:

- app launches
- database file is created
- migration runs exactly once
- no product behavior changes yet

### Slice 2: JSON Importer

Create:

- `HolyMigrationService.swift`

Implement:

- import existing JSON snapshot to DB
- record import marker
- validate row counts

Acceptance for slice 2:

- existing `v0.1` users upgrade without losing sessions/templates/archive

### Slice 3: Event Append on Session Create/Archive/Relaunch

Implement:

- session rows
- lifecycle event rows
- archive projection rows

Acceptance for slice 3:

- event history starts accumulating for the most important workflows

## Risks

Primary risks:

### 1. Over-normalizing too early

If the schema is too abstract in `v0.2`, development slows down for little gain.

Countermeasure:

- keep launch spec and some projections as JSON blobs where appropriate

### 2. Supervisor split destabilizes UI

If the split happens before durable persistence exists, the product can regress badly.

Countermeasure:

- sequence DB first, supervisor second

### 3. Event spam without semantic clarity

If every tiny mutation becomes an event, the ledger becomes noisy and expensive.

Countermeasure:

- only record domain-significant transitions in `v0.2`

## v0.2 Acceptance Checklist

`v0.2` is ready to ship when all are true:

- SQLite is the durable source of truth
- session event ledger exists and is queryable
- archived sessions and templates survive restart through the DB
- current `v0.1` state imports forward cleanly
- `HolyWorkspaceStore` is materially smaller and no longer the persistence/supervisor choke point
- the current `v0.1` operator workflow still works
- versioned read-only compatibility views exist for future external consumers
- the product is structurally ready for telemetry, budgets, and multi-surface modes in later releases

## After v0.2

Once this plan is complete, the next release should be `v0.3`:

- structured runtime telemetry
- cost and budget intelligence
- richer event-driven inspector surfaces

That work becomes realistic only after `v0.2` lands.
